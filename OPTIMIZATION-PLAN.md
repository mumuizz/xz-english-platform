# 项目优化方案

## 1. 当前项目概览

项目由两部分组成：

- `client/`：React + Vite + TypeScript 前端
- `server/`：Express + Prisma + SQLite 后端

当前代码已经具备基础功能，但存在几个典型问题：

- 部分页面请求次数偏多，前端存在重复拉取数据的情况
- 后端路由中有较多重复逻辑，维护成本偏高
- 单词导入与统计逻辑存在数据漂移风险
- 文案和编码历史问题较多，后续需要统一清理
- 缺少自动化测试、Lint、格式化、性能基线

---

## 2. 已识别的优化方向

### 2.1 前端优化

1. **减少重复请求**
   - `Ebbinghaus` 页面原先为了展示词库数量，会对每个词库分别发一次请求
   - 在词库切换时又重复加载统计数据，导致无效请求增多

2. **减少重复计算**
   - 当前复习中的单词数据存在重复 `JSON.parse`
   - 这会增加渲染期开销，也让代码更难维护

3. **页面职责拆分**
   - 部分页面文件过大，UI、请求、数据转换、交互逻辑混在一起
   - 建议后续拆成 hooks、service、presentational components

4. **统一接口层**
   - 当前 `client/src/utils/api.ts` 只做了基础 token 注入
   - 后续可增加统一错误提示、重试策略、请求取消、类型约束

### 2.2 后端优化

1. **路由重复逻辑收敛**
   - `words.ts` 中存在重复的参数解析、ID 校验、导入构建逻辑
   - 已开始收敛，但其他路由也可继续整理

2. **统计接口聚合**
   - 原先前端为了词库统计自己拼装请求
   - 更合理的方式是由后端直接返回聚合后的统计结果

3. **数据一致性**
   - 单词新增/删除与用户 `totalWords` 更新原先分开执行
   - 异常时可能出现计数不同步
   - 已改为事务处理

4. **导入逻辑稳定性**
   - 单词大小写、首尾空格未统一处理，容易造成重复数据
   - `import-sample` 旧逻辑对“新增”和“更新”统计不够准确
   - 已在本轮先修正

### 2.3 数据层优化

1. **建议增加唯一约束**
   - 建议后续在 Prisma 中为 `Word` 增加复合唯一键，例如：`@@unique([userId, word])`
   - 这样能从数据库层彻底杜绝重复单词

2. **建议补索引**
   - 复习场景高频依赖：
     - `userId`
     - `vocabSet`
     - `nextReview`
   - 后续可考虑为高频查询字段建立索引

### 2.4 工程化优化

1. **补齐测试**
   - 后端至少补路由级 smoke test
   - 前端至少补关键页面和数据转换逻辑测试

2. **补齐质量工具**
   - ESLint
   - Prettier
   - Type-safe API types

3. **补齐运行说明**
   - 当前构建环境依赖未完全安装时，无法直接执行 `tsc`
   - 建议补充更清晰的启动与安装文档

## 3. 本次已完成的第一批优化

### 已完成

#### A. 后端 `words` 路由优化

- 新增 `GET /api/words/stats`
  - 由后端聚合返回各词库总量与待复习数量
- 收敛了单词标准化逻辑
  - 导入与新增时统一 `trim + lowercase`
- 增强了 ID 解析与参数校验
- 新增/删除单词时改为事务更新
  - 同步维护 `User.totalWords`
- 重写了 `import-batch` / `import-sample`
  - 统一新增和更新流程
  - 修复统计不准确问题
  - 降低重复代码

#### B. 前端 `Ebbinghaus` 页面优化

- 将“每个词库单独请求一次”改为“一次聚合请求”
- 将词库统计请求从“切换词库就重新拉”改为“初始化和导入后刷新”
- 将当前复习单词的解析结果做 memo 化，避免重复 `JSON.parse`
- 重构页面状态组织，便于后续继续拆分
- 清理了本页一部分用户可见提示文案

## 4. 分阶段优化建议

### Phase 1：已完成

- 聚合词库统计接口
- 收敛复习页请求
- 修复单词导入计数问题
- 收敛部分后端重复逻辑

### Phase 2：建议下一步做

1. **数据库约束与索引**
   - 为 `Word` 增加唯一键
   - 为高频查询字段增加索引

2. **接口返回类型统一**
   - 统一后端返回结构，例如：
     - `{ success: true, data }`
     - `{ success: false, error }`

3. **前端页面拆分**
   - 把 `Ebbinghaus` / `Listening` / `Reading` 这类大页面拆成：
     - page
     - hooks
     - api service
     - display components

### Phase 3：稳定性与质量

1. 引入 ESLint / Prettier
2. 增加后端接口测试
3. 增加前端关键页面测试
4. 清理历史编码乱码文案

### Phase 4：性能与体验

1. 懒加载大型页面
2. 列表虚拟化
3. 图片资源压缩与缓存策略
4. 复习流程的操作反馈优化

## 5. 当前阻塞项

本地执行 `npm run build` 时，当前环境提示：

- `client` 下的 `tsc` 未找到

这通常表示当前本地依赖尚未安装完整，或 `node_modules` 未准备好。

建议先执行：

```bash
npm run install:all
```

然后再验证：

```bash
npm run build
```
